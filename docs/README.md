# 開発・運用ドキュメント

最終更新日: 2026-07-19

このディレクトリは、2026年夏の関西旅行Webアプリの要件、設計、運用および重要な技術判断を管理する。利用者向けに公開する旅行資料は `public/docs/` に置き、本ディレクトリには開発・保守に必要な情報のみを置く。

## 文書体系

| 文書 | 目的 | 主な読者 |
|---|---|---|
| [requirements.md](requirements.md) | 目的、対象範囲、機能要件、非機能要件、受入基準を定義する | プロダクト所有者、実装者、テスト担当 |
| [architecture.md](architecture.md) | システム境界、構成要素、実行時フロー、外部依存、品質上の判断を定義する | 実装者、保守担当 |
| [data-design.md](data-design.md) | 静的旅程、進捗データ、識別子、整合性、保持方針を定義する | 実装者、DB管理者 |
| [ui-design.md](ui-design.md) | 画面責務、情報設計、操作原則、アクセシビリティを定義する | UI実装者、テスト担当 |
| [operations.md](operations.md) | デプロイ、監視、障害対応、旅行前後の運用を定義する | 運用担当、リポジトリ所有者 |
| [adr/](adr/) | 重要なアーキテクチャ判断とトレードオフを記録する | 将来の実装者、レビュー担当 |

## 文書の優先順位

仕様に矛盾がある場合は、次の順で解釈する。

1. 明示的に承認された最新の要件およびADR
2. 現在のソースコードとデータベーススキーマ
3. 本ディレクトリのその他の設計文書
4. ルート `README.md`
5. `public/docs/` の利用者向け資料

コードが文書と異なる場合は、コードを無条件に正とせず、意図した変更か文書更新漏れかを確認する。

## 更新規則

- 機能、データモデル、外部サービス、セキュリティ、運用手順を変更する場合は、同一の変更単位で関連文書も更新する。
- 機能要件には `FR-*`、非機能要件には `NFR-*`、運用要件には `OPS-*` の識別子を付ける。
- 既存要件の意味を変更するときは、受入基準と関連テストも見直す。
- 採用済みの重要な設計判断を変更する場合は、既存ADRを書き換えず、新しいADRで置き換えを記録する。
- 旅行日程は `src/data/trip.ts`、利用者向け文書は `public/docs/`、進捗状態はSupabaseを正本とする。

## 対象外

この文書体系は、汎用旅行サービスや商用SaaS向けの完全な規制対応文書ではない。2026年8月20日から25日までの家族旅行を安全かつ確実に支援し、短期運用後に停止またはアーカイブできることを目的とする。
