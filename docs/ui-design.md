# UI・UX設計書

最終更新日: 2026-07-20

## 1. UI目標

旅行中の利用者が、屋外・移動中・片手操作・不安定な通信環境でも、現在必要な情報を短時間で誤解なく取得できることを目標とする。

実用性を最優先としつつ、家族旅行の期待感と非日常感を損なわない。情報階層を崩さず、風景、タイポグラフィ、余白、質感、動きを限定的に用いて、旅行専用アプリとしての固有性を持たせる。

## 2. 設計原則

1. 現在必要な情報を最上位に置く。
2. 次の行動、時刻、場所を一つの視線範囲で確認可能にする。
3. 確定情報と候補情報を混同させない。
4. 色だけで意味を伝えない。
5. 破壊的操作や重大変更は不用意に実行させない。
6. 通信失敗とデータ不在を区別する。
7. 外部サービスへ移る操作を明示する。
8. 家族全員が説明なしでも主要機能を利用できる語彙を用いる。
9. 読み込み中でも既存の静的情報を隠さない。
10. オフライン情報が古い可能性を明示する。
11. 装飾は判断を妨げない範囲に限定し、重要情報のコントラストを優先する。
12. 交通手段を汎用の列車・航空機アイコンだけで表現せず、区間、駅名、時刻、番線、予約状態を主要な視覚要素にする。

## 3. Scenic Cinema デザインシステム

### 3.1 方向性

「旅行映画の冒頭」と「実用的な行程カード」を組み合わせる。ホーム画面では風景を大きく扱い、その上または直下に現在必要な行動を配置する。

- 雰囲気: 明るい家族旅行、夕景、旅の始まり
- 主色: 深い青緑
- 副色: コーラル、淡い金色、アイボリー
- 背景: 完全な白ではなく、紙を思わせる暖色系の淡色
- タイポグラフィ: 本文はゴシック、旅行タイトルのみ明朝体
- 角丸: 主要カード 22〜29px
- 影: 境界線だけに依存せず、低彩度で柔らかい多層シャドウを使う

### 3.2 ビジュアル

- ホーム上部に関西の夕景を抽象化したローカルSVGを使用する。
- 外部画像の読み込みを必須にせず、オフライン時も主要な世界観を維持する。
- 風景は情報の背景に限定し、文字には暗いグラデーションを重ねて可読性を確保する。
- ノイズやぼかしは低強度に留める。

### 3.3 アイコン方針

- 新幹線、電車、航空機などを安価な汎用アイコンで代替しない。
- 交通情報は発着地、時刻、所要時間、番線、予約状態、経路線で表現する。
- アイコンを使う場合も補助的用途に限定し、アイコンだけで意味を伝えない。
- 下部ナビゲーションは番号と文字ラベルを組み合わせ、説明性を保つ。

### 3.4 動き

- カード初期表示の短いフェードアップのみを基本とする。
- `prefers-reduced-motion` を尊重する。
- 動きを重要情報の唯一の伝達手段にしない。

## 4. 情報階層

優先順位は以下とする。

1. 今やること
2. 次の予定
3. 時刻と場所
4. 地図への導線
5. 当日の全旅程
6. 交通・宿泊
7. 資料
8. 天気等の補助情報
9. チャット等の付加機能

## 5. 画面責務

### 5.1 ホーム

目的: 現在と次の行動を即時把握し、旅行の期待感を提示する。

表示:

- 旅行期間と主な訪問地域
- 現在日時・旅行日
- 今やること
- 次の予定
- 主要な警告
- 天気の要約
- 当日旅程への導線

ホームに全情報を詰め込まず、判断に必要な最小情報を優先する。

### 5.2 旅程

目的: 全日程または選択日の予定を順序付きで確認する。

要件:

- 日付単位に分割する。
- 別行動グループを明確に分ける。
- 合流後の予定を識別する。
- 時刻未定を確定時刻のように見せない。
- 確定度を文字と視覚表現の双方で示す。

### 5.3 交通・宿泊

目的: 移動と滞在に必要な情報へ集約して到達する。

- 交通手段のアイコンを主役にせず、区間と時刻を主役にする。
- 宿泊は施設名、地域、日程、状態をカード内で確認可能にする。
- 公開リポジトリであるため、予約番号、QRコード、個人電話番号等は表示しない。
- 必要な機微情報は別の安全な共有手段へ誘導する。

### 5.4 資料

目的: `public/docs/` の利用者向け資料を閲覧する。

- ファイル名ではなく人が理解できるタイトルを表示する。
- 読み込み失敗時は再試行方法を示す。
- 未キャッシュ資料はオフライン時に開けない可能性を示す。

### 5.5 チャット

目的と保存方針が確定していない場合、旅行の中核機能として扱わない。誤った自動回答を旅程の正本とみなさせない。

## 6. ナビゲーション

画面下部の固定ナビゲーションを使用する。

| 項目       | ラベル     |
| ---------- | ---------- |
| ホーム     | ホーム     |
| 旅程       | 旅程       |
| 交通・宿泊 | 交通・宿泊 |
| 資料       | 資料       |
| チャット   | チャット   |

要件:

- 現在地を視覚的かつARIA上で識別可能にする。
- 番号だけでなく文字ラベルを表示する。
- セーフエリアを考慮する。
- 主要操作領域を44px以上とする。

## 7. カード設計

旅程イベントカードは以下を一貫した順序で表示する。

1. 時刻
2. タイトル
3. 場所または経路
4. 確定度
5. 説明
6. 地図リンク

表示ルール:

- 時刻未定は「時刻未定」と明示する。
- 終了時刻がない場合、存在しない値を推測しない。
- 長い説明は主要情報を押し下げすぎないよう整理する。
- 地図リンクは外部遷移を文字と記号で明示する。
- `→` を含む場所は発着地を分割し、経路線として表示してよい。

## 8. 状態表現

### 8.1 確定度

| 値          | 日本語 | 意味               |
| ----------- | ------ | ------------------ |
| `confirmed` | 確定   | 原則として実施予定 |
| `candidate` | 候補   | 選択肢の一つ       |
| `undecided` | 未定   | 判断が必要         |

## 9. フィードバック

### 読み込み中

- 静的旅程は可能な限り即時表示する。
- 天気等の外部情報のみ個別に読み込み状態を表示する。
- 画面全体を不要にブロックしない。

### オフライン

- オフラインであることを表示する。
- 閲覧中の情報がキャッシュであり、最新同期済みとは限らないことを示す。
- 利用不能なオンライン機能を無反応にしない。

## 10. レスポンシブ方針

- 320px以上のモバイルを最小対象とする。
- 基準画面は6.1インチiPhone相当の幅393px、高さ852pxとする。
- 最大コンテンツ幅を430pxに制限し、タブレット・PCでは中央に表示する。
- 主要レイアウトは単一カラムを基本とする。
- 日付選択以外の主要操作に横スクロールを使用しない。
- 長い地名・説明の折り返しを許可する。

## 11. アクセシビリティ

- 文書構造に適切な見出し階層を用いる。
- ナビゲーションに `aria-label` を付ける。
- アイコンのみのボタンにはアクセシブルネームを付ける。
- フォーカス表示を消さない。
- フォーム要素に明示的ラベルを付ける。
- エラー表示はスクリーンリーダーで認識できるようにする。
- タップ対象を44px以上とする。
- 背景と文字の十分なコントラストを確保する。
- 動きの大きい演出を必須情報伝達に使わない。

## 12. UI受入チェック

- iPhone SE相当幅で横スクロールがない。
- 393px幅でホーム上部、現在行動カード、日付選択が崩れない。
- 下部ナビゲーションがセーフエリアと重ならない。
- 今と次の予定がホーム上部にある。
- 状態と確定度が文字で分かる。
- 長いタイトル・場所・説明が崩れない。
- 交通画面で汎用交通アイコンに依存していない。
- Supabase失敗時にも旅程を読める。
- オフライン状態を誤認しない。
- キーボードで主要操作へ到達できる。
- 200%拡大でも主要情報が欠落しない。
