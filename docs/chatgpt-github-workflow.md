# ChatGPT GitHub変更ワークフロー

最終更新日: 2026-07-22

## 1. 目的

ChatGPTが`harutoyama/2026-kansai-trip`を安全かつ再現可能に変更し、検証、Pull Request、マージ、デプロイ確認まで実施するための標準手順を定義する。

この文書はリポジトリ直下の`AGENTS.md`を補足する。矛盾する場合は、より厳しい安全要件を優先する。

## 2. 標準フロー

1. 最新`main`の完全commit SHAを取得して`BASE_SHA`として固定する。
2. 変更対象と関連識別子をリポジトリ全体から検索する。
3. 対象ファイルを`add`、`update`、`delete`へ分類する。
4. リスクを低・中・高に分類する。
5. 変更を実装し、必要なテストを追加または修正する。
6. 差分、旧参照、機密情報、影響範囲を確認する。
7. 低リスクは`main`へ直接commitする。中・高リスクは作業ブランチへcommitし、Pull Requestを作成する。
8. CIで`test`、`lint`、`build`を実行する。
9. CI結果、ログ、PR差分、head SHAを確認する。
10. 中リスクは問題がなければChatGPTがマージする。高リスクはユーザー承認後にマージする。
11. `main`のデプロイ結果と公開反映を確認し、未確認事項を報告する。

## 3. 基準コミットと競合防止

- ファイル取得は可能な限り`BASE_SHA`をrefとして行う。
- commit作成後、PRのhead SHAを固定してレビューする。
- マージ時は期待するhead SHAを指定し、レビュー後に追加commitが入った場合はマージしない。
- `main`が先に進んだ場合、競合や意味的な不整合を確認し、必要なら最新`main`へ変更を積み直す。
- force push、強制ref更新、`main`履歴の書換えは行わない。

## 4. ブランチとPull Request

中・高リスク変更では、次の命名を基本とする。

```text
chatgpt/<変更内容の短い識別子>
```

Pull Request本文には以下を記載する。

- 目的
- 変更内容
- 変更ファイル
- リスク分類
- 実施した検証
- 自動テストで確認できない事項
- ロールバック方法

一つの論理的変更は、一つのPRと可能な限り一つの原子的commitにまとめる。

## 5. テスト設計

### 単体テスト

純粋関数、変換処理、バリデーション、状態遷移など、入出力を独立して検証できる処理に使用する。

### コンポーネント・結合テスト

Reactコンポーネント、Hook、フォーム操作、画面遷移、複数モジュールの連携に使用する。Supabase等の外部サービスは、目的に応じてモックまたはテスト用境界を用いる。

### 回帰テスト

不具合修正では、可能な限り修正前に失敗し修正後に成功するテストを追加する。

### テストを追加しない場合

文書のみの変更、既存テストで同一挙動が十分に担保される変更、テスト追加が実質的な価値を持たない変更では、理由を明示した上で追加しないことがある。

## 6. 必須CI

PR用CIは、原則として以下を実行する。

```bash
npm ci
npm test
npm run lint
npm run build
```

- Node.jsのバージョンはPagesデプロイと合わせる。
- `package-lock.json`を使用して依存関係を固定する。
- CIで利用しない権限は付与しない。
- CI失敗時はジョブとステップのログを確認し、原因を特定して修正commitを追加する。
- CIを無効化、迂回、または成功扱いにしてマージしてはならない。

## 7. レビュー項目

マージ前に以下を確認する。

- 依頼内容と差分が一致している。
- 計画外のファイルが変更されていない。
- 不要な全体整形やlockfile更新がない。
- 旧識別子、旧ルート、旧props、旧型参照が残っていない。
- TypeScript型と実行時データ構造が一致している。
- Supabaseスキーマ、RLS、Realtime、クライアント実装が整合している。
- Secrets、service role key、予約番号、連絡先等が含まれていない。
- `test`、`lint`、`build`が成功している。
- 自動検証できない事項が明示されている。

## 8. マージ方針

- 中リスク変更は、CI成功と差分レビュー完了後、ChatGPTがマージしてよい。
- 高リスク変更は、CI成功後もユーザーの明示的承認を待つ。
- マージ時はレビュー済みのPR head SHAを指定する。
- 原則としてsquash mergeを使用し、一つの論理的変更を`main`上の一つのcommitにまとめる。
- branch protectionやrequired status checksが設定されている場合は、それらを迂回しない。

## 9. デプロイ確認

`main`マージ後は、GitHub Pagesワークフローについて以下を確認する。

- build jobの成功
- deploy jobの成功
- 対象commitがデプロイされたこと
- 公開URLが応答すること

UI、PWA、Realtime、端末間同期等、GitHub Actionsだけで保証できない事項は、未確認として報告し、必要な手動確認手順を示す。

## 10. ロールバック

- マージ前はPRを閉じ、`main`を変更しない。
- マージ後はrevert commitまたはrevert PRを作成し、履歴を書き換えない。
- DBやRLS等の外部変更には、事前に逆向きmigrationまたは復旧手順を用意する。
- 復旧後も`test`、`lint`、`build`およびデプロイ確認を行う。
