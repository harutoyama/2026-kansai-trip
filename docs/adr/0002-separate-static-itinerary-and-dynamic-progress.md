# ADR-0002: 静的旅程と動的進捗を分離する

- Status: Accepted
- Date: 2026-07-19
- Decision owners: Haru

## Context

旅程本体は旅行前に計画・レビューされ、変更頻度が低い。一方、未開始、実行中、完了、遅延、中止等の進捗は旅行中に複数端末から更新される。すべてをDBへ保存すると障害時に旅程自体を失う可能性があり、すべてを静的ファイルにすると端末間同期ができない。

## Decision drivers

- 外部サービス障害時にも旅程を閲覧する
- 旅程変更をGitでレビューする
- 進捗だけを複数端末で即時共有する
- データモデルと運用を単純化する
- オフライン閲覧を可能にする

## Considered options

1. 旅程と進捗の双方をSupabaseに保存
2. 旅程と進捗の双方をクライアント内だけに保存
3. 旅程を静的データ、進捗をSupabaseに保存
4. GitHubリポジトリを実行時DBとして直接更新

## Decision

旅程本体を `src/data/trip.ts` に保持し、進捗をSupabase `event_progress` に保存する。両者は安定した `event_id` で結合する。

## Rationale

旅程をビルド成果物に含めることで、Supabaseが停止しても閲覧できる。進捗だけを小さなDBモデルに限定することで、Realtime同期と匿名更新のリスク範囲を縮小できる。Gitによる旅程レビューとDBによる実行時更新の責務も明確になる。

## Consequences

### Positive

- 静的旅程の可用性が高い
- DBスキーマが小さい
- 旅行前の変更履歴をGitで追跡できる
- オフライン時も旅程を表示できる
- Supabase未設定でも主要閲覧機能を利用できる

### Negative

- 旅程変更には再ビルド・再デプロイが必要
- `event_id` 変更で進捗との対応が失われる
- 静的・動的データをマージする実装が必要
- 古いPWAキャッシュと最新DB進捗の組み合わせが生じ得る

## Validation

- Supabaseを無効化しても全旅程を閲覧できること
- 同じ `event_id` の進捗がイベントへ適用されること
- ID重複をテストで検出できること
- 旅程変更後も既存イベントIDが維持されること
