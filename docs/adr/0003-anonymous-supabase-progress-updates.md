# ADR-0003: 進捗に対する匿名Supabase更新を許可する

- Status: Accepted
- Date: 2026-07-19
- Decision owners: Haru

## Context

家族4人が短期旅行中に利用し、アカウント作成やログイン操作は避けたい。一方で複数端末から進捗状態を共有したい。GitHub Pagesは静的配信であり、独自認証サーバーを持たない。

## Decision drivers

- 非技術利用者がログインなしで使える
- 複数端末で進捗を共有できる
- 実装と運用を短期間で完了できる
- 保存対象を非機密の進捗へ限定する
- 旅行終了後に権限を停止できる

## Considered options

1. Supabase Authを導入
2. 共有パスワードを独自実装
3. 匿名ロールに限定的なinsert/updateを許可
4. 端末内保存のみとし同期しない

## Decision

Supabase anonロールに `event_progress` のselect、insert、updateを許可する。deleteは許可しない。DB制約によりID長、状態、遅延値、メモ長を制限する。予約情報等の機微情報は保存しない。

## Rationale

利用者数と運用期間が限定され、保存データも非機密の旅行進捗に限定されるため、認証導入の操作・実装コストよりも利用容易性を優先する。リスクはデータ項目の最小化、DB制約、旅行後の権限停止で抑制する。

## Consequences

### Positive

- ログイン不要
- 家族端末から即時更新可能
- Realtime同期を利用できる
- アカウント管理が不要

### Negative

- URLとSupabase情報を得た第三者が進捗を変更できる
- 更新者を識別・監査できない
- URL秘匿は正式なアクセス制御ではない
- 荒らしや誤操作への耐性が限定的
- 旅行後に停止作業を忘れると匿名更新が継続する

## Risk controls

- 保存項目を進捗関連だけに限定する
- 機微情報を保存しない
- 状態をenumで限定する
- 遅延値を0〜1440分に制限する
- メモを500文字以下に制限する
- delete権限を付与しない
- 旅行中に不審更新を確認する
- 2026年9月30日までに匿名insert/updateを停止する

## Validation

- 未認証ブラウザから正しい値を保存できること
- 不正状態、過大遅延、長すぎるメモが拒否されること
- deleteが拒否されること
- 匿名更新ポリシーを停止する手順が運用文書に存在すること
