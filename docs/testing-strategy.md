# テスト戦略

最終更新日: 2026-07-22

## 目的

利用者が実際に操作するフォーム、ボタン、画面遷移を、純粋関数のテストだけでなくReactコンポーネントおよび実ブラウザでも検証する。特に不具合修正では、修正前に失敗し修正後に成功する回帰テストを追加する。

## テスト層

### 単体テスト

純粋関数、変換処理、バリデーション、日時処理、状態遷移を対象とする。対象ファイルと同じディレクトリへ`*.test.ts`として配置する。

### コンポーネント・結合テスト

React Testing Libraryを使用し、画面の描画、フォーム入力、ボタン操作、Hookとの連携を検証する。Supabaseは本番へ接続せず、テスト用Hookまたはモックへ置き換える。対象画面と同じディレクトリへ`*.test.tsx`として配置する。

### ブラウザE2Eテスト

PlaywrightのChromiumを使用し、Viteで起動したテスト専用画面を実ブラウザから操作する。テスト専用画面はインメモリデータを使用し、本番Supabase、認証情報、家族の実データへ接続しない。テストは`e2e/`へ配置する。

## 不具合修正時の基準

- 操作不能、保存不能、画面遷移不能などの不具合には、利用者の操作手順を再現するコンポーネントテストを追加する。
- DOM環境だけでは検出しにくい問題には、PlaywrightによるブラウザE2Eテストを追加する。
- 同一挙動を既存テストが十分に保証している場合は重複テストを追加しない。
- テストを追加できない場合は、Pull Request本文へ理由と手動確認手順を記載する。

## 必須コマンド

```bash
npm test
npm run lint
npm run build
npm run test:e2e
```

Pull Request CIでは上記をすべて実行する。Playwrightレポートは失敗時にGitHub Actions artifactとして保存する。

## 手動確認が必要な項目

以下は自動テストだけでは完全に保証できないため、変更内容に応じて実機で確認する。

- GitHub Pagesへの反映とPWAキャッシュ更新
- iPhoneおよびAndroid端末でのタップ操作
- Supabase Realtimeによる端末間同期
- 実データに対するRLS、認証、権限
- フォントサイズ、折返し、スクロール、セーフエリア

## テストデータと機密情報

テストには架空の固定データのみを使用する。Supabase URL、anon key、service role key、旅行の認証PIN、予約番号、連絡先をテストコード、snapshot、artifactへ含めない。
