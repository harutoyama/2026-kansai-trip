# 要件定義書

最終更新日: 2026-07-20
対象リリース: 2026年夏の関西旅行本番運用版

## 1. 目的

本システムは、2026年8月20日から25日までの家族4人による関西旅行について、旅程、移動、宿泊、資料、現在の進行状況および補助情報をスマートフォンから一元的に確認できるようにする。

旅行中は、情報探索の時間を最小化し、「今何をするか」「次にどこへ向かうか」「予定が変更されたか」を同行者間で共有できることを最優先とする。

## 2. 利用者と利用環境

### 2.1 主利用者

- Haru
- 同行する家族3名
- 技術知識を持たない利用者を含む
- 認証アカウントの作成を要求しない

### 2.2 主利用環境

- iPhoneまたはAndroidスマートフォン
- モバイル回線または宿泊施設等のWi-Fi
- Safari、Chrome等の標準的なモバイルブラウザ
- GitHub Pages上の公開URL
- 通信が不安定または一時的に利用不能となる状況

## 3. スコープ

### 3.1 対象

- 日別・グループ別の旅程表示
- 現在および次の予定の提示
- 交通・宿泊情報の表示
- 利用者向け資料の表示
- 天気情報の取得
- Google Mapsへの遷移
- PWAとしての利用
- 静的旅程のオフライン閲覧

### 3.2 対象外

- 航空券、鉄道、ホテル等の予約・変更
- 決済
- 汎用旅行計画サービス
- 不特定多数向けのユーザー登録・認証
- 厳密なアクセス制御を前提とする機密情報管理
- アプリ内ターンバイターンナビゲーション
- 旅行終了後の恒久的な商用運用
- 位置情報の継続追跡または同行者監視

## 4. 用語

| 用語 | 定義 |
|---|---|
| 旅程イベント | 移動、食事、観光、宿泊等の一つの予定 |
| 確定度 | `confirmed`、`candidate`、`undecided` のいずれか |
| 静的旅程 | アプリのソースコードに含まれる予定本体 |
| 縮退動作 | 外部サービスが利用できない場合に、利用可能な機能だけで運用を継続すること |

## 5. 機能要件

### FR-001 日別旅程表示

システムは、2026年8月20日から25日までの旅程を日付単位で表示しなければならない。

**受入基準**

- 各日に日付、日数、概要、主なエリアが表示される。
- イベントは予定順に読み取れる。
- イベントのタイトル、時刻、場所、確定度が存在する場合は表示される。

### FR-002 グループ別旅程表示

システムは、同行者が別行動する時間帯について、グループごとの予定を区別して表示しなければならない。

**受入基準**

- 2026年8月20日の11:20より前について、晴と家族3人のいずれかを選択UIで切り替え、予定を混同せず確認できる。
- 11:20以降の共通予定は選択中のグループにかかわらず別セクションとして確認できる。
- 確定済み交通では発着時刻、発着番線、便・列車名、座席を表示できる。

### FR-003 現在および次の予定

システムは、現在日時と旅程データから、現在実行中とみなされる予定および次の予定を提示しなければならない。

**受入基準**

- 現在時刻が予定時間内の場合、対象予定が「今やること」として表示される。
- 予定間の時間帯では、直後の予定が次の予定として表示される。
- 旅行期間外または該当予定がない場合は、誤った予定を表示せず説明を表示する。

### FR-006 Supabase未構成時の利用

Supabase接続情報が設定されていない場合でも、システムは静的旅程の閲覧、地図遷移、利用者向け資料の閲覧および可能な範囲の天気表示を提供しなければならない。

### FR-007 地図連携

利用者は、場所を持つ旅程イベントからGoogle Mapsの検索画面を開けなければならない。

**受入基準**

- 地図リンクはホテル、観光施設等、地図確認に意味があるイベントへ明示的に設定する。
- 駅・列車・航空便等の交通イベントには自動生成の地図リンクを表示しない。
- `mapsQuery` が設定されたイベントのみ検索語として用いる。
- 外部画面へ遷移することが分かる表示を備える。

### FR-008 天気表示

システムは、旅行判断の補助として、現在地または予定地点に対応する天気情報を表示できなければならない。

**受入基準**

- 外部APIの応答が得られた場合、気温等の主要情報を表示する。
- API失敗時に旅程表示を阻害しない。
- 天気情報を交通機関の運行保証や安全保証として扱わない。

### FR-009 利用者向け資料

システムは、`public/docs/` に置かれた旅行資料をアプリ内から参照できなければならない。

### FR-010 PWA・オフライン閲覧

利用者は、対応ブラウザにおいて本システムをPWAとして利用でき、事前に取得済みの静的旅程を通信不能時にも閲覧できなければならない。

**制約**

- Realtime同期、最新天気、未キャッシュの資料、外部地図は通信不能時に利用できない可能性がある。
- オフライン時に表示される情報が最新とは限らないことを明示する。

## 6. 非機能要件

### NFR-001 モバイル優先

幅320px以上の画面で、主要情報と主要操作を横スクロールなしで利用できること。

### NFR-002 操作対象

主要なボタン、リンク、選択UIは、原則として44px以上の操作領域を確保すること。

### NFR-003 情報取得時間

利用者がホーム画面を開いてから、現在または次の予定を原則3操作以内で確認できること。

### NFR-004 可用性と障害分離

Supabase、Open-Meteo、Google Mapsのいずれかが利用不能でも、他の独立した機能まで連鎖的に利用不能にしないこと。

### NFR-005 データ整合性

- `event_id` は静的旅程と動的進捗を結合する安定識別子とする。
- 公開後に同一イベントの `event_id` を変更しない。

### NFR-006 セキュリティ・プライバシー

- 予約番号、QRコード、電話番号、自宅住所、本人確認情報を公開リポジトリへコミットしない。
- クライアントへ配布されるSupabase anon keyを秘密情報として扱わず、RLSを実質的な権限制御境界とする。
- 匿名更新可能であることと、そのリスクを利用者・運用担当が理解できるよう文書化する。
- 機微情報は別の非公開経路で共有する。

### NFR-007 アクセシビリティ

- 主要ナビゲーションに意味のあるラベルを付ける。
- 色だけで確定度を伝えない。
- フォーカス可能な要素をキーボードで操作できるようにする。
- エラーを視覚情報だけに依存せず伝える。
- 文字と背景のコントラストを屋外利用に耐える水準とする。

### NFR-008 保守性

- 画面、共通UI、静的データ、外部連携、型定義を責務別に分離する。
- 旅程計算等の純粋ロジックは自動テスト可能な形にする。
- 仕様変更と文書更新を同一変更単位で行う。

### NFR-009 デプロイ再現性

`main` ブランチから、GitHub Actionsによりテスト、ビルド、GitHub Pages配信を再現できること。

### NFR-010 短期運用終了性

旅行終了後、匿名書き込み停止、公開停止またはリポジトリアーカイブを実施できること。

## 7. 優先順位

| 優先度 | 内容 |
|---|---|
| Must | 旅程表示、現在・次予定、状態更新、モバイル表示、障害時の静的閲覧 |
| Should | 端末間Realtime同期、PWA、天気、資料閲覧 |
| Could | 実績時刻、遅延分、メモの高度な編集 |
| Won't for this release | 認証、予約、決済、汎用旅行作成機能 |

## 8. 総合受入条件

本番運用開始前に、少なくとも以下を満たすこと。

1. 全6日分の旅程を実機で確認できる。
2. 8月20日の別行動がグループ別に表示される。
3. 状態更新が同一端末と別端末に反映される。
4. Supabaseを無効化しても静的旅程が表示される。
5. 通信遮断後にキャッシュ済み旅程が表示される。
6. 地図リンクが正しい検索対象を開く。
7. 外部API失敗時に画面全体が停止しない。
8. 公開ファイルに機微情報が含まれない。
9. GitHub Actionsのテストとビルドが成功する。
10. 旅行終了後の停止担当と実施日が決まっている。
