# 運用設計書

最終更新日: 2026-07-19

## 1. 運用目的

旅行前の準備、旅行中の安定利用、障害時の復旧、旅行終了後の安全な停止を、リポジトリ所有者が再現可能な手順で実施できるようにする。

## 2. 環境

| 環境 | 用途 |
|---|---|
| ローカル開発 | 実装、テスト、プレビュー |
| GitHub Actions | CI、ビルド、Pagesデプロイ |
| GitHub Pages | 本番配信 |
| Supabase | 本番進捗保存・Realtime |
| 利用端末 | 家族のスマートフォン、PWA |

## 3. 必要な設定

GitHub Actions Secrets:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

注意:

- これらはビルド成果物へ含まれ得るため、anon key自体を秘密の認証情報として扱わない。
- データ保護はSupabase RLSで実施する。
- service role keyは絶対にフロントエンドやGitHub Pagesへ渡さない。

## 4. デプロイ手順

1. 変更内容と関連文書を更新する。
2. ローカルで依存関係を導入する。
3. `npm test` を実行する。
4. `npm run lint` を実行する。
5. `npm run build` を実行する。
6. ローカルプレビューで主要画面を確認する。
7. `main` へpushする。
8. GitHub Actionsのbuildとdeploy成功を確認する。
9. 公開URLで新しいコミットが反映されたことを確認する。
10. PWA端末でキャッシュ更新を確認する。

## 5. リリース前チェックリスト

### データ

- [ ] 2026年8月20日から25日まで全日が存在する
- [ ] 8月20日の別行動グループが正しい
- [ ] イベントIDに重複がない
- [ ] 時刻と場所が最新である
- [ ] 候補・未定事項が確定情報として表示されない
- [ ] 機微情報が含まれない

### 機能

- [ ] ホームに現在・次の予定が表示される
- [ ] 全日程を閲覧できる
- [ ] 地図リンクが適切な場所を開く
- [ ] 状態を変更できる
- [ ] 別端末へ同期される
- [ ] 天気失敗時も旅程を閲覧できる
- [ ] Supabase未設定時も静的機能が利用できる
- [ ] オフラインでキャッシュ済み旅程を閲覧できる

### 実機

- [ ] 家族全員の端末でURLを開ける
- [ ] 必要端末でPWAをホーム画面へ追加した
- [ ] 文字サイズとタップ領域に問題がない
- [ ] 充電・モバイル回線・ブラウザ設定を確認した
- [ ] 代替の旅行情報を端末または紙に保存した

## 6. 監視

本システムは短期・小規模運用のため、常時監視基盤は導入しない。旅行期間中は以下を定期確認する。

- GitHub Pagesへアクセスできるか
- 最新デプロイが成功しているか
- Supabaseへ読み書きできるか
- 端末間同期が動作するか
- Open-Meteo取得失敗が継続していないか
- 不審な進捗変更がないか

## 7. 障害対応

### OPS-001 サイトが開かない

1. 他の回線または端末で確認する。
2. GitHub StatusおよびActions実行結果を確認する。
3. URLとGitHub Pages設定を確認する。
4. インストール済みPWAまたは保存済み代替資料を使う。
5. 復旧を待たず、旅行判断は公式交通情報・保存資料で継続する。

### OPS-002 最新版が表示されない

1. ブラウザを再読み込みする。
2. アプリを完全終了して再起動する。
3. PWAの更新待ち状態を確認する。
4. 必要ならサイトデータまたはPWAを削除して再取得する。
5. 表示中の版が最新かコミット内容と照合する。

### OPS-003 進捗が同期されない

1. 端末の通信状態を確認する。
2. Supabase環境変数がビルドへ設定されているか確認する。
3. Supabaseプロジェクトの稼働状態を確認する。
4. `event_progress` とRLSポリシーを確認する。
5. Realtime publicationを確認する。
6. 同期不能中は口頭またはメッセージで共有し、静的旅程を利用する。

### OPS-004 状態更新に失敗する

1. エラー表示を確認する。
2. 再試行する。
3. `event_id`、状態値、遅延値、メモ長が制約内か確認する。
4. SupabaseログとRLSを確認する。
5. 必要なら旅行中は状態更新を使わず、旅程閲覧のみ継続する。

### OPS-005 天気が表示されない

- 旅程機能は継続利用する。
- 気象庁等の公式情報を別途確認する。
- 天気表示を安全判断の唯一の根拠にしない。

### OPS-006 不審な更新

1. 家族へ変更事実を確認する。
2. Supabaseの匿名insert/updateポリシーを一時停止する。
3. 正しい状態へ修正する。
4. 必要に応じてURL公開停止またはSupabase接続を切る。
5. 機微情報流出がないことを確認する。

## 8. バックアップと代替手段

旅行開始前に以下を用意する。

- 全日程のPDFまたはスクリーンショット
- 航空・鉄道・宿泊情報の非公開コピー
- 緊急連絡先
- 予約情報
- 家族間の連絡手段
- 必要に応じた紙の控え

公開アプリを唯一の情報源にしない。

## 9. 変更管理

旅行直前および旅行中の変更は、影響範囲を限定する。

- 日程変更は対象イベントだけを更新する。
- 依存関係の大規模更新やUI全面改修を避ける。
- 本番中にDBスキーマを破壊的変更しない。
- 緊急修正でもテストと公開確認を省略しない。
- 変更後、家族へ更新内容を共有する。

## 10. 旅行終了後

旅行終了後、遅くとも2026年9月30日までに以下を判断・実施する。

- [ ] 進捗データを保存するか決定
- [ ] 必要ならデータをエクスポート
- [ ] Supabase匿名insert/updateを無効化
- [ ] 不要データを削除
- [ ] GitHub Pages公開継続を判断
- [ ] リポジトリのアーカイブを判断
- [ ] Secretsを削除するか判断
- [ ] 文書に実運用上の知見を追記
- [ ] 機微情報が残っていないことを再確認
